# 改善候補（備忘録）

現状の仕様ではなく、今後の改善候補を記録するメモです。仕様そのものは [`specification.md`](./specification.md) を参照してください。

> 優先度・粒度はあくまで目安です。着手時に再評価してください。

## 中優先度

### Context の再レンダー最適化
- 巨大な単一 Context に state・派生値・`actions` をまとめて配布しているため、いずれかの変更で広範囲が再描画される。
- `actions` が未メモ化（毎回新しい関数）になっている。
- 対策案: Context を state / actions / UI 状態などに分割する、`useCallback` / `useMemo` で Provider value を安定化する、描画コストの高い画面・行コンポーネントを `React.memo` 化する。

### `updateSet` の全ワークアウト走査
- `useWorkoutActions.ts` の `updateSet` / `updateSetIntensity` が全ワークアウトを `map` で走査している。
- セット入力のたびに全 `workouts` / 全 `sets` の新しいオブジェクトを作るため、履歴が増えたときの入力遅延につながりやすい。
- 対策案: `currentWorkoutId` または `setId -> workoutId` のインデックスを使い、対象ワークアウトだけを更新する。

### 画面内派生値のメモ化・セレクタ化
- `HomeScreen` の合計値、`HistoryScreen` の `visibleHistory` / `trainedDates` / `trainingDayByDate` / カレンダーセルなどが、画面再描画ごとに再計算される。
- データ量が少ない間は問題になりにくいが、ローカル保存の履歴が蓄積するとカレンダー・履歴表示の再描画コストが上がる。
- 対策案: 純粋な集計ロジックを `selectors/fitLogSelectors.ts` に寄せ、必要な箇所で `useMemo` する。

### コード重複の解消
- `plannedPartsForDate` が `selectors/fitLogSelectors.ts` と `screens/HistoryScreen.tsx` に重複。
- 曜日ラベル（`['日','月',...]`）が複数箇所に定義。
- カレンダーセルの組み立て、確認ダイアログ、履歴リスト作成、HEX → rgba 変換などの小ロジックが画面ごとに散在。
- 対策案: セレクタ/ユーティリティ/共通コンポーネントへ集約。

### `weight` / `recordValue` の `string | number` 型
- 入力途中は文字列、計算時は数値という二重の責務を 1 つの型が担っている。
- `number()` / `isBlank()` で吸収しているが、保存値・入力値・計算値の境界が曖昧になり、移行やバリデーションを増やすほど扱いにくくなる。
- 対策案: 保存値は数値または空値に正規化し、入力中の文字列状態は UI 層に閉じ込める。互換性維持のため `storage.ts` に移行・正規化を追加する。

### 保存データのスキーマバージョン管理
- 現在は `catalogVersion` で種目マスタ追補を管理し、保存データ全体は `normalizeState` で後方互換を吸収している。
- 今後、`WorkoutSet` や `Workout` など保存形式を大きく変える場合、どの移行が適用済みかを判定しづらくなる。
- 対策案: `State` に `schemaVersion` を追加し、`storage.ts` に段階的な migration 関数を用意する。

### ダイアログのアクセシビリティ
- フォーカストラップ、Escape での閉じる、バックドロップ外側クリックでの閉じるが未実装。
- `HomeScreen` の削除確認・カレンダー、`PresetListScreen` / `PresetEditScreen` の削除確認、`SelectScreen` の追加/編集ダイアログで実装が分かれている。
- 対策案: 共通ダイアログコンポーネントにフォーカス制御、Escape、外側クリック、初期フォーカス、閉じた後のフォーカス復元を実装。

### メニューとドラッグ操作のアクセシビリティ
- `HistoryScreen` の設定メニューは開閉できるが、Escape / 外側クリック / フォーカス移動の扱いが限定的。
- `SelectScreen` の種目並び替えはポインタ操作中心で、キーボード操作で並び替える導線がない。
- 対策案: メニュー用の共通挙動を用意し、ドラッグ対象には上下移動ボタンやキーボードショートカットを併設する。

### ネストしたインタラクティブ要素
- `HomeScreen` のカードが `role="button"` で、その内側に削除ボタンを内包している（ネストしたインタラクティブ要素）。
- 対策案: カードの構造を見直し、操作要素のネストを解消。

### インポート時のデータ保護
- JSON インポートは現在の `state` を全置き換えするが、事前確認や自動バックアップはない。
- 誤ったファイルを選んだ場合、直前の状態に戻すにはブラウザや OS 側の手段に頼る必要がある。
- 対策案: インポート前に確認ダイアログを出す、または現在の state を自動でエクスポート/退避してから置き換える。

### テスト範囲の拡充
- 現在のテストは `storage.test.ts` と `utils.test.ts` が中心。
- ワークアウト操作、プリセット投入、部位編集、計画プレビュー、インポートなど、ユーザー操作に近いフック/セレクタのテストが薄い。
- 対策案: `selectors/fitLogSelectors.ts` と主要 action hook の純粋化しやすいロジックからテストを追加する。

## 低優先度

- `isUnstartedWorkout` がセット数 5 に依存している（マジックナンバー）。
- 削除操作の Undo（取り消し）が無い。
- トーストが連続発火で上書きされる（キューイングが無い）。
- カレンダー実装がホーム/履歴で別々（共通化の余地）。
- `tsconfig` の厳格化（`noUncheckedIndexedAccess` など）。
- 依存バージョンの `"latest"` 指定（`react` / `vite` / `typescript` など）をピン留めする。
- `PartEditScreen` など一部画面で、表示用の小コンポーネントと操作ロジックが同じファイルにまとまっている（規模が増えたら分割余地）。
- `buildPartRecentLabels` が部位ごとに全 `workouts` を filter/sort している（履歴が増えた場合は事前インデックス化の余地）。

- 設定メニュー（開閉・外側クリック・Escape・フォーカス復元）
- カレンダーグリッド（月移動・日付セル・trained/selected/today 表示）
- 記録単位トグル（回数/秒数）※現状は `SelectScreen` 内に閉じている
- 部位タブ / 曜日ピッカー（色つき選択状態・横スクロール・押しやすいサイズ）
