# 改善候補（優先順位順）

現状の仕様ではなく、今後の改善候補を記録するメモです。仕様そのものは [`specification.md`](./specification.md) を参照してください。

2026-06-20 時点の実装を再調査し、データ保護・不具合リスク・回帰防止・操作性・性能・保守性の順で優先順位を付けています。完了した項目は残していません。

## 高優先度

### 1. 部位削除後に部位が復活する状態の解消

- `deletePart` は `state.parts` と同じ部位の `trainingPlans` だけを削除するが、`buildOrderedParts` は過去の `workouts` と `trainingDays` からも部位を復元する。
- 種目がなくても過去記録に同じ部位がある場合、削除成功のトースト後に部位が一覧へ再表示されるため、現在の UI 上の「削除」と実際の挙動が一致しない。
- 対策案: 部位を「表示設定から非表示にする」操作として設計し直すか、履歴を保持したまま非表示状態を保存できるデータ構造を追加する。過去記録まで変更・削除する場合は影響範囲を明示して確認する。
- `usePartActions` と `buildOrderedParts` の組み合わせをテストし、過去記録がある部位の扱いを固定する。

### 2. インポート時の検証・確認・ロールバック

- JSON インポートは `normalizeState` を通した直後に現在の state を全置換する。
- 現在データの自動退避、置換前の確認、件数差分のプレビュー、部分的に除外された不正データの通知がないため、誤ファイルや一部破損したバックアップで記録を失う可能性がある。
- 対策案: インポート前に現在データを自動バックアップし、種目・ワークアウト・プリセットなどの件数と警告を確認してから確定する。確定後も直前データへ戻せるようにする。
- 完全に妥当なデータ、部分破損、別形式、巨大ファイル、キャンセル、復元のテストを追加する。

### 3. 主要操作の回帰テストと CI の品質ゲート

- 現在の 68 テストは `storage`、`utils`、セレクタ、種目並び替えが中心で、`useWorkoutActions`、`usePresetActions`、`usePartActions`、画面遷移、インポート確定などの主要フローを直接検証していない。
- GitHub Pages の workflow は `npm run build` だけを実行し、lint とテストに失敗してもデプロイを止められない。
- 対策案: 状態更新ロジックを純粋関数へ切り出して、開始・入力・終了・再開・削除・プリセット投入・部位編集・インポートを優先してテストする。workflow では `npm run lint`、`npm test`、`npm run build` をデプロイ前に実行する。

## 中優先度

### 4. 複数タブ・複数ウィンドウからの上書き防止

- 保存先が単一の `localStorage` キーで、別タブの更新を検知する `storage` イベントや競合判定がない。
- 古い state を開いたままのタブで操作すると、後から保存した側が新しい記録を丸ごと上書きする可能性がある。
- 対策案: 保存データに更新日時またはリビジョンを持たせ、外部更新を検知したら再読み込み・退避・競合確認を行う。少なくとも古い state の無警告上書きは防止する。

### 5. 未開始ワークアウト判定をセット数から分離

- `isUnstartedWorkout` は「セットがちょうど 5 件で、すべて未入力」の場合だけ未開始と判定する。
- 空セットを追加・削除して件数が 5 以外になると、入力がなくても終了時の未開始確認・除外対象にならず、空のワークアウトが保存される。
- 対策案: セット数ではなく「入力済みセットが 1 件もない」で判定する。初期表示セット数は別の定数または生成処理の責務として扱い、0 件や 5 件超もテストする。

### 6. ダイアログ・メニュー・複合操作のアクセシビリティ

- 複数のダイアログとドロワーで、フォーカストラップ、Escape、初期フォーカス、閉じた後のフォーカス復元が統一されていない。
- `HistoryScreen` の設定メニューは外側クリック・Escape・矢印キー移動に未対応。
- `HomeScreen` の `role="button"` のカード内に削除ボタンがあり、インタラクティブ要素が入れ子になっている。
- `SelectScreen` の種目並び替えはポインタ操作中心で、キーボード用の移動手段がない。
- 対策案: 共通 Dialog / Menu の挙動を用意し、カード構造を分離する。並び替えには上下移動ボタンまたはキーボード操作を併設する。

### 7. Context の購読範囲と actions の安定化

- 単一 Context が state、派生値、UI 状態、`actions` をまとめて配布し、各画面は Context 全体を購読している。
- `useFitLog` と各 action hook が毎レンダーで関数・`actions` オブジェクトを作り直すため、セット入力やトースト表示でも広い範囲が再描画される。
- 対策案: 保存 state、UI state、actions の Context を分割するか selector 付き Context を採用する。更新関数を `useCallback`、Provider value を `useMemo` で安定化し、計測後に負荷の高い行を `React.memo` 化する。

### 8. セット更新時の全ワークアウト走査

- `updateSet` と `updateSetIntensity` は対象セットを事前検索した後、全 `workouts` と全 `sets` を再度 `map` している。
- 入力のたびに履歴全体のオブジェクトを作り直すため、記録が増えるほど入力遅延と Context 再描画の影響が大きくなる。
- 対策案: `currentWorkoutId`、`setId -> workoutId` の索引、または対象ワークアウトを特定する更新ヘルパーを使い、対象だけを更新する。

### 9. 保存データのスキーマバージョンと段階的 migration

- `catalogVersion` はスターター種目の追補専用で、保存データ全体は現在の `normalizeState` が一括で後方互換を吸収している。
- 保存形式の変更が増えると、どの移行を適用したか、古い形式をいつまで扱うかが判別しにくくなる。
- 対策案: `State` に `schemaVersion` を追加し、バージョンごとの migration を順番に適用してから現行形式を検証する。各世代の fixture で移行テストを行う。

### 10. 入力値と保存値の型分離

- `WorkoutSet.weight` / `recordValue` は、入力途中の文字列と計算用の数値を同じ `string | number` で表している。
- `number()`、`isBlank()`、表示・単位変換関数が境界を吸収しており、保存値の保証やバリデーションを追加するほど扱いが複雑になる。
- 対策案: 保存値は数値または明示的な空値へ正規化し、入力中の文字列は画面ローカル state に閉じ込める。既存の `localStorage` には migration を用意する。

## 低優先度

### 11. 履歴増加時の集計インデックス

- `buildPartRecentLabels` は部位ごとに全ワークアウトを `filter` / `sort` し、ホーム画面の集計や履歴画面にも配列走査が残る。
- 現在のデータ量では問題になりにくいため、実データで計測してから日付・種目・部位別の索引や 1 回の走査へ置き換える。

### 12. UI フィードバックの堅牢化

- トーストは連続発火すると前の内容を上書きする。
- 削除操作は確認ダイアログが中心で、削除後の Undo はない。
- 対策案: 通知キューを導入し、ワークアウトやプリセットなど復元しやすい削除から Undo を追加する。

### 13. 依存関係・型チェック設定の固定

- `react`、`vite`、`typescript` などが `package.json` で `"latest"` のため、lockfile を作り直す際に意図しないメジャー更新が入りやすい。
- `strict` は有効だが、`noUncheckedIndexedAccess` などの追加余地がある。
- 対策案: 主要依存を意図した範囲へ固定し、Dependabot 等で更新を可視化する。厳格化はエラー量を確認しながら段階的に行う。

### 14. 重複 UI と大規模画面の整理

- ホームと履歴に別々のカレンダー表示があり、ダイアログのマークアップも複数画面に重複している。
- `HistoryScreen`、`HomeScreen`、`DetailScreen` は表示と操作ロジックが大きく、変更時の確認範囲が広い。
- 対策案: 見た目を無理に統一せず、日付セル・月移動・Dialog の振る舞いなど安定した単位から共通化する。画面は意味のあるセクション単位で分割する。
